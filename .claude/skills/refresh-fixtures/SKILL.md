# Refresh and review test fixtures

This project's unit tests run offline against recorded JSON/XML/HTML
fixtures under `tests/fixtures/<source>/`, generated by actually calling each
live API once. Those fixtures can silently go stale: an upstream source can
change its response shape, add/remove a field, or alter status semantics,
and the offline unit suite will keep passing forever against the old
snapshot, giving false confidence that the client still works.

This isn't only a concern when adding a new source — during ordinary work
this session, re-running the regeneration script also picked up real,
unprompted upstream drift in three *already-existing* sources (Catalogue of
Life, Mushroom Observer, PBDB) with no code change on this project's side at
all.

## Two different scripts, two different jobs

- **`python tests/fixtures/check_fixtures.py`** — read-only. Fetches fresh
  data from every live API and diffs it against the currently saved
  fixtures, without overwriting anything. Use this to *find out* whether
  anything changed.
- **`python tests/fixtures/regenerate_fixtures.py`** — fetches fresh data and
  **overwrites** the saved fixture files. Use this once you've decided the
  new live response is what the fixtures should reflect.

Both make real network calls to every configured source and will take
roughly the same amount of time; `check_fixtures.py` is the safer one to run
first since it can't accidentally bake a bad/transient response into the
fixtures.

## Reviewing what changed

After regenerating, treat the diff as real signal, not just noise to
`git add`:

```bash
git status --short tests/fixtures/
git diff tests/fixtures/<source>/
```

For each changed fixture file, ask: is this a **meaningless timestamp/version
bump** in the upstream data (safe to accept as-is), or a **structural change**
(a renamed field, a new required field, a changed status string) that the
corresponding client in `scripts/apis_pipe/<source>.py` needs to be updated
to handle? Run `pytest -m "not integration" tests/scripts/apis_pipe/test_<source>.py`
after regenerating that source's fixtures to confirm the existing
`_compile_accepted`/`_compile_synonyms` logic still parses the new shape
correctly, not just that the fixture file was written.

## Credentialed sources

Sources needing credentials (Tropicos, MycoBank, GenBank) will silently skip
themselves during regeneration if the required environment variables aren't
set locally (each fetcher prints a `SKIPPED` line rather than failing) — so
"the script ran clean" doesn't guarantee every source was actually checked.
Confirm your local `.env` has the relevant credentials before trusting a
clean run to mean "everything is up to date."

## Committing

Fixture files are ordinary tracked files, not generated build artifacts —
`git add` and commit the ones that genuinely changed. Don't let regenerated
fixtures for sources you weren't investigating sit as unrelated noise in an
unrelated commit; split them out if a change is unrelated to what you're
actually working on.
